# 003: 3層キャッシュアーキテクチャ

## Status

Accepted

## Context

リモートサーバーから画像をストリーミング表示するアプリで、レスポンス速度・メモリ消費・ディスク使用量・ユーザー意図 (ダウンロード) のバランスを取る必要がある。

## Decision

3層キャッシュ + お気に入り (メタデータのみ) の構成を採用。

| 層 | 保存先 | 排出 | 用途 |
|---|---|---|---|
| L1 | メモリ | LRU 自動 (~10枚) | 現在表示中 + 近傍ページ |
| L2 | ディスク | LRU 自動 (500MB〜5GB) | セッション中のキャッシュ |
| L3 | ディスク | 手動トグル | ユーザーが明示的にDLした作品 |
| お気に入り | JSON | 手動トグル | URL + メタデータのみ (画像なし) |

`CacheManager.get(key)` が L1 → L2 → L3 の順に検索し、ヒット時は上位層に昇格。

## Alternatives

### 2層 (メモリ + ディスク)

- L3 がないとユーザーのダウンロード意図を表現できない。LRU で意図的に保存した画像が消える

### ディスクキャッシュのみ

- メモリキャッシュがないとページめくりのたびにディスク I/O が走る。特にスワイプ時のラグが大きい

## Consequences

- **Good**: L1 でページめくりが即座。L2 で同一セッション中の再表示がディスクから即座。L3 でユーザーの保存意図を永続化
- **Good**: 各層が独立したポリシーで排出。L2 は LRU で自動管理、L3 はユーザー操作のみ
- **Good**: キー命名 (`thumb:` / `full:`) で同一アイテムのサムネイルとフル画像を分離
- **Bad**: L2 のサイズ管理 (atomic write、メタデータ JSON) の実装が必要
- **Bad**: L3 の作品単位ダウンロード (複数ページ、ZIP、PDF) で種別ごとの保存ロジックが必要
